## 2. Markdown 정책 해석 (현재 이슈 포인트)

기획서 기준:
- 카드 설명은 "경량 Markdown 지원"
- MVP 허용: 굵게/기울임, 문단/줄바꿈, 헤딩, 리스트, 링크
- v1.1 확장 고려: 체크리스트, @멘션, 카드 참조 링크

근거:
- `docs/syncflow/실시간_경량_협업_보드_제품기획서_v1.md`  
  (3.5.1, 3.5.2 섹션)

### 패키지 사용 여부

- 문서 어디에도 "반드시 특정 패키지 사용" 요구는 없음.
- 따라서 `flutter_markdown`은 "필수"가 아니라 "선택"이다.
- 결론:
  - 전체 Markdown(헤딩/링크/강조 등)을 빠르게 맞추려면 `flutter_markdown` 계열 패키지 채택이 유리
  - 제한 기능(예: 체크리스트 + 멘션)만 유지하려면 커스텀 렌더도 가능

### 현재 코드와 정책 정합성

- 현재 구현은 커스텀 렌더 기반이며 체크리스트/@멘션 중심
- 즉, 기획서의 "MVP Markdown 기본 집합"보다 "v1.1 확장 요소"를 먼저 일부 반영한 상태
- 또한 길이 제한은 기획서 예시(3000)와 다르게 현재 4000

정리:
- 의도적으로 부분 기능만 갈 경우 문서에 "현재 지원 문법"을 별도 명시해야 혼선이 줄어든다.

## 3. 문서 간 범위 차이 (구현 시 주의)

### 3.1 컬럼 이벤트
- WS 스펙 문서에는 컬럼 이벤트가 "1차 옵션/2차 이관 가능"으로 표기
- 현재는 REST 기반 컬럼 관리가 구현되어 있음
- 의미: 컬럼은 "기능 있음", "WS 실시간 이벤트는 별도 확장"으로 해석

### 3.2 멤버 초대
- 초대 설계서는 "2차 적용" 문구가 있으나, 현재 코드에는 초대 코드 생성/참가가 존재
- 의미: 문서의 적용 시점보다 구현이 선행된 상태

### 3.3 동기화 기준
- 동기화 문서는 최신 구현 기준(ACK 매칭, 부분 패치, invalidate 폴백 등)을 상세히 담고 있음
- 실시간 이슈 수정 시 해당 문서를 1순위로 참고
