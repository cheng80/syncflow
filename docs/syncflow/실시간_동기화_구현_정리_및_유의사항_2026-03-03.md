# SyncFlow 실시간 동기화 구현 정리 및 유의사항

작성일: 2026-03-03  
대상 범위: Flutter(`lib/*`) + FastAPI(`fastapi/app/ws/*`, `fastapi/app/api/cards.py`)

## 1. 문서 목적

본 문서는 SyncFlow의 카드 실시간 동기화 구현 상태를 정리한다.

- 현재 동작 구조(클라이언트/서버 역할)
- 이벤트별 동기화 흐름
- 최근 수정 사항(깜빡임/되돌아감/동기화 누락 대응)
- 유지보수 시 주의사항
- 검증 체크리스트

## 2. 현재 동기화 아키텍처 요약

### 2.1 기본 원칙

- **서버 권위(Server Authoritative)**: 최종 상태는 서버가 결정
- **클라이언트 낙관적 업데이트(Optimistic UI)**: 이동 시 즉시 UI 반영
- **`req_id` 기반 ACK 매칭**: 본인 요청 확정 타이밍 추적
- **부분 패치 우선**: 생성/수정/삭제는 전체 재조회 대신 캐시 패치
- **필요 시 invalidate 폴백**: 캐시 없음/복구 필요 시 전체 재조회

### 2.2 주요 상태 Provider

파일: `lib/vm/board_detail_notifier.dart`

- `boardDetailProvider(boardId)`: 서버 상세 스냅샷(FutureProvider)
- `boardDetailCacheProvider(boardId)`: WS 부분 패치 반영용 캐시
- `optimisticCardMovesProvider(boardId)`: 이동 낙관적 상태(`cardId -> {columnId, position}`)
- `pendingMoveReqIdsProvider(boardId)`: ACK 대기 중 이동 요청 `req_id` 집합

## 3. 이벤트별 동기화 동작

## 3.1 카드 이동 (`CARD_MOVE` / `CARD_MOVED`)

### 클라이언트

파일: `lib/view/board_detail_screen.dart`

1. 드래그 시 `_displayCards` 즉시 재배치
2. 낙관적 상태를 해당 컬럼 카드 전체에 반영
3. WS 연결 시 `CARD_MOVE` 전송 (`before_card_id`, `after_card_id`, `req_id`)
4. `CARD_MOVED` 수신 시:
   - `req_id`가 `pendingMoveReqIds`에 있으면 본인 ACK로 간주
   - pending/retry 상태 정리 후 **부분 패치로 캐시 반영** (`column_cards` 사용)
5. ACK timeout(2초) 시 1회 재전송, 재시도 후에도 미수신이면 롤백 + invalidate

### 서버

파일: `fastapi/app/ws/handlers.py`

1. 멤버/카드/컬럼 유효성 검증
2. `before_card_id`/`after_card_id` 기준으로 삽입 인덱스 계산
3. 대상 컬럼 카드 순서를 `0, 1000, 2000...`으로 재정규화
4. `CARD_MOVED` 브로드캐스트(`req_id`, `column_cards`, `board_version` 포함)

### 주의

- `before_card_id`는 이동 카드의 **이전 이웃**, `after_card_id`는 **다음 이웃**
- 해석 반대 시 카드가 엉뚱한 위치로 이동
- 본인 `CARD_MOVED`를 무시하면 서버 확정 반영이 지연되거나 불일치 가능

---

## 3.2 카드 생성 (`CARD_CREATE` / `CARD_CREATED`)

### 클라이언트

- 추가 시트(`_AddCardSheet`)에서 WS 연결이면 `CARD_CREATE` 전송
- 응답 대기 없이 시트 닫기
- `CARD_CREATED` 수신 시 `boardDetailCacheProvider`에 카드 추가(부분 패치)

### 서버

- `CARD_CREATE` 처리 후 `CARD_CREATED` 브로드캐스트(`req_id` 포함)
- 신규 카드는 **맨 위 삽입** 규칙 적용:
  - `position = MIN(active positions) - 1000`
  - 비어 있으면 `0`

동일 규칙은 REST `POST /v1/cards`에도 적용됨 (`fastapi/app/api/cards.py`).

---

## 3.3 카드 수정 (`CARD_UPDATE` / `CARD_UPDATED`)

### 클라이언트

- 상세 모달 저장 시 WS 연결이면 `CARD_UPDATE` 전송
- `CARD_UPDATED` 수신 시 해당 카드만 patch 적용(부분 패치)

### 서버

- patch 필드 검증/업데이트
- `CARD_UPDATED` 브로드캐스트(`req_id` 포함)

### 주의

- UI diff 비교에 `id/position`만 쓰면 제목/설명 변경이 화면에 반영되지 않음
- 현재 `_ColumnView._listEquals`는
  - `id`, `position`, `columnId`, `title`, `description`, `priority`, `status`
  를 비교하도록 수정됨

---

## 3.4 카드 삭제(아카이브) (`CARD_ARCHIVE` / `CARD_ARCHIVED`)

### 클라이언트

- 상세 모달 삭제 시 WS 연결이면 `CARD_ARCHIVE` 전송
- `CARD_ARCHIVED` 수신 시 캐시에서 카드 제거(부분 패치)

### 서버

- status를 `archived`로 변경
- `CARD_ARCHIVED` 브로드캐스트(`req_id` 포함)

---

## 4. 깜빡임/되돌아감 이슈와 대응

## 4.1 이동 후 이전 상태로 잠깐 돌아가는 현상

원인:

- 낙관적 반영 직후 무조건 전체 `invalidate` + 로딩 동안 낙관적 상태 초기화

대응:

- 이동은 `req_id` ACK 기반으로 확정 타이밍 제어
- ACK 대기 중(`pendingMoveReqIds` not empty)에는 낙관적 상태 유지

## 4.2 `position` 충돌로 원위치되는 현상

원인:

- 동일 `position` 전송 시 서버 정렬(`position, id`)에서 순서 반전 가능

대응:

- 이동 API를 `before/after` 중심으로 전환
- 서버가 최종 인덱스를 계산하고 position 재정규화

## 5. Riverpod 사용 시 유의사항 (중요)

## 5.1 빌드 중 provider 상태 변경 금지

실제 발생 에러:

- `Tried to modify a provider while the widget tree was building`

원인:

- `build`/`ref.listen` 동기 구간에서 `state = ...` 수행

현재 대응:

- `ref.listen` 내부 provider 쓰기는 `WidgetsBinding.instance.addPostFrameCallback`으로 지연
- `build`에서는 읽기만 수행

---

## 6. invalidate 사용 규칙

### 전체 invalidate를 유지하는 경우

- `CARD_RESTORED` 수신 시(아직 부분 패치 미구현)
- 캐시가 없는 상태에서 WS patch 이벤트 수신 시
- 이동 ACK timeout/재시도 실패 시 복구용

### 부분 패치만 수행하는 경우

- `CARD_CREATED`
- `CARD_MOVED`
- `CARD_UPDATED`
- `CARD_ARCHIVED`

## 7. 서버/클라이언트 계약(Contract)

## 7.1 WS 요청에 `req_id` 사용

- 클라이언트는 변경 요청에 `req_id` 포함 권장
- 서버는 성공 이벤트에 동일 `req_id`를 포함해 에코

## 7.2 공통 필드 유의

- `board_id` 누락 시 필터링/반영 불가
- `card_id`, `column_id`, `position` 타입은 정수 보장 필요
- `board_version`은 서버 기준 최신 상태 판단에 사용

## 7.3 board_version 정책

- 서버는 카드 변경 이벤트에 `board_version`을 포함
- `board_version` 값은 `boards.updated_at` 기반 epoch ms
- 클라이언트는 **구버전 이벤트(incoming < current)** 를 무시
- REST/WS 카드 변경 시 `boards.updated_at`을 갱신하여 버전 전진

## 8.1 Flutter 로그

- `[카드이동] WebSocket ACK 수신(req_id=...)`

- `[동기화] 구버전 이벤트 무시: ...`
- `서버 데이터 수신 → 낙관적 초기화`
- 이벤트 수신 타입(`CARD_CREATED`, `CARD_UPDATED`, ...)

## 8.2 서버 로그/코드 확인

- `fastapi/app/ws/handlers.py`의 payload 구성(`req_id` 포함 여부)
- `before/after` 인덱스 계산 로직
- 생성 시 top insertion(`MIN(position)-1000`)

## 9. 테스트 체크리스트

## 9.1 단일 클라이언트

1. 카드 생성 시 맨 위 삽입
2. 카드 수정(제목/설명) 즉시 반영
3. 카드 삭제 즉시 반영
4. 카드 이동 후 되돌아감 없는지 확인

## 9.2 다중 클라이언트(권장)

1. A에서 생성 → B 즉시 반영
2. A에서 수정 → B 즉시 반영
3. A에서 삭제 → B 즉시 반영
4. A에서 이동 → B 순서 정확히 반영
5. 동시에 이동 시 최종 서버 상태 일관성 확인
6. 한쪽 네트워크 끊김 후 복구 시 자동 rejoin + 스냅샷 복구 확인
